# Teardown v2 — 4단계 작업 지시서 (LLM 폴백 완성 브리핑)

> VS Code Claude Code용. 이 문서를 읽고 그대로 구현. 채팅(설계)에서 확정.
> 대상: src/Teardown.jsx. 충돌 방지 위해 VS Code에서만 수정.

---

## 목표

룰이 못 잡는 새 문제(예: T5 gemma JSONDecodeError 런타임 버그)를 만났을 때,
사용자가 자기 LLM(Claude/Gemini 챗)에 붙여넣을 **"완성 브리핑"**을 개선한다.

**현재 buildBriefing의 한계 = 반쪽짜리.** 지금은 [워크플로 구조] + [에러 로그]만 담는다.
환경 정보도, 양자화 진단 결과도, 룰이 이미 발견한 힌트도 안 담겨서, LLM이 맥락을 절반만 받는다.

**4단계 = 브리핑에 다음을 추가:**
1. 사용자 환경 (GPU/torch/CUDA) — 2단계에서 입력받은 env
2. 양자화 진단 결과 — 3단계 quantWarnings (fp8/Ampere 경고)
3. 룰이 이미 잡은 처방 요약 — buildPrescription 결과 (LLM이 중복 안 하게)

이렇게 하면 "디버깅 병목 = 컨텍스트 조립"이라는 이 도구의 핵심 가치가 브리핑에서 완성된다.

## 구현 상세

### 1. reportToContext(report, env) — env 추가
현재 함수(약 253줄)에 env 파라미터 추가. 끝에 환경 블록 추가:
```js
function reportToContext(report, env) {
  // ... 기존 packs/models/muted/port ...
  const lines = [
    `전체 노드: ${report.totalNodes} / 커스텀 pack: ${report.customPackCount}`,
    `패키지·버전:\n${...}`,
    `참조 모델·자산:\n${...}`,
    `이식 위험 값:\n${...}`,
    `비활성 노드: ${muted.join(", ") || "없음"}`,
  ];
  // env 있으면 추가
  if (env && (env.gpu || env.torch || env.cuda)) {
    lines.push(`사용자 환경: GPU=${env.gpu||"?"} / torch=${env.torch||"?"} / CUDA=${env.cuda||"?"}`);
  }
  return lines.join("\n");
}
```
- runAiDiagnosis에서도 reportToContext(report, env) 호출하도록 (env 넘기게). env 없으면 기존과 동일.

### 2. buildBriefing(report, errlog, env) — 환경 + 양자화 진단 + 룰 처방 요약 추가
현재 함수(약 711줄) 개조:
```js
function buildBriefing(report, errlog, env) {
  const ctx = reportToContext(report, env);
  const qw = quantWarnings(report.models, env?.gpu);  // 3단계 함수 재사용
  const rx = buildPrescription(report, env?.gpu);       // 룰 처방 (양자화 경고 포함됨)
  const L = [];
  L.push(`아래는 ComfyUI 워크플로의 구조 분석 + 환경 + (있으면)에러 로그입니다.`);
  L.push(`이 정보를 종합해 이 워크플로의 구체적 노드·모델·환경에 맞춰 진단하고, 해결을 단계별로 알려주세요.`);
  L.push(`확신 없는 부분은 솔직히 밝히고, 없는 URL·파일을 지어내지 마세요.`);
  L.push(``);
  L.push(`## 워크플로 구조 (대상: ${report.source})`);
  L.push("```"); L.push(ctx); L.push("```");
  L.push(``);
  // ★ 양자화 진단 (환경 입력됐고 경고 있을 때만)
  if (qw.length) {
    L.push(`## 이미 발견된 양자화 호환성 문제 (Teardown 룰)`);
    for (const w of qw) {
      L.push(`- ${w.file}: ${w.quant} 형식 → ${w.gen} GPU에서 ${w.support===false?"미지원":"부분지원"} → ${w.alt} 권장`);
    }
    L.push(``);
  }
  // ★ 룰이 이미 잡은 처방 요약 (LLM이 중복 진단 안 하게)
  if (rx && rx.length) {
    L.push(`## Teardown이 이미 제시한 처방 (참고 — 중복 진단 말고 이 다음을 도와주세요)`);
    rx.forEach((step, i) => L.push(`${i+1}. ${step.title}`));
    L.push(``);
  }
  L.push(`## 에러 로그`);
  L.push("```");
  L.push(errlog.trim() || "(에러 로그 없음 — 구조·환경만 보고 점검해 주세요)");
  L.push("```");
  L.push(``);
  L.push(`---`);
  L.push(`Generated by Teardown · 이 브리핑을 그대로 붙여넣고 엔터를 누르세요.`);
  return L.join("\n");
}
```

### 3. copyBriefing 호출부 (약 938줄) — env 넘기기
`buildBriefing(report, errlog)` → `buildBriefing(report, errlog, env)`

### 4. ★ "LLM 분석 프롬프트" 버튼 활성화 조건 변경 (약 1483줄)
현재: `disabled={!errlog.trim()}` — 에러 로그 없으면 비활성.
문제: 4단계 후엔 환경+양자화 진단만으로도 브리핑이 가치 있음. 에러 로그 없어도 환경 입력됐으면 브리핑 줄 만함.
변경: `disabled={!errlog.trim() && !(env.gpu || env.torch || env.cuda)}`
→ 에러 로그 OR 환경 입력 둘 중 하나라도 있으면 활성화.
- opacity·cursor 조건도 같은 식으로 변경 (errlog.trim() || env 입력됨 → 1, 아니면 0.4).
- 단 "AI 정밀 진단"(td-cta, doAiDiagnosis, 1478줄) 버튼은 에러 로그 필수 유지 (그건 로그 진단이므로). 바꾸는 건 "LLM 분석 프롬프트"(copyBriefing) 버튼만.
- 안내 문구도 살짝: 에러 로그 없이 환경만으로도 복사 가능함을 자연스럽게 (placeholder나 sub문구). 과하지 않게.

## 핵심 규칙 (회귀 방지)
- env 없으면: 환경 블록·양자화 블록 다 안 들어감 → 기존 브리핑과 거의 동일 (회귀 없음).
- quantWarnings/buildPrescription은 3단계에서 이미 만든 함수 재사용. 새로 안 만듦.
- 브리핑은 여전히 **API 호출 0** (사용자가 자기 챗에 붙여넣는 용도). 비용 0, 배포본 OK.

## 완료 기준
1. 환경(3090) 입력 + fp8 모델 JSON에서 "LLM 분석 프롬프트" 복사 → 브리핑에 환경·양자화 경고·룰 처방 요약이 다 포함됨.
2. 환경 미입력 시 → 브리핑에 환경/양자화 블록 없음 (기존과 동일, 회귀 없음).
3. ★ 에러 로그 없이 환경만 입력해도 "LLM 분석 프롬프트" 버튼이 활성화됨 (에러도 환경도 둘 다 없으면 비활성).
4. 브리핑은 API 호출 안 함 (복사만).
5. npm run build 에러 없음.
