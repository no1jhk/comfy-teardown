# Teardown — 전략 점검: compatibility.json 보강 방법론 + 실증 계획 + 다음 가능성 (2026-06-27)

> 사용자 지적: "진도가 많이 나간 것 같지 않다, 가시적 결과가 약하다.
> comp.json 보강 방법론을 심도깊게. 이후 해볼 것 더 찾아라. 실제 PC에서 빨간줄 검증할 거다."
> → 두세 번 고민한 현실적 답.

## 0. 먼저 — "진도가 약해 보이는" 이유 (정직하게)

맞는 지적이다. 오늘 한 일(양자화 진단, 4층 데이터소스, UX)은 **내부 구조 개선**이라 화면에서 극적으로 안 보인다.
가시적이려면 "이 도구로 실제 막힌 걸 풀었다"는 사례가 필요하다. → 그래서 사용자의 실증 계획(아래 §2)이 정확한 다음 스텝이다.
**지금까지는 "도구를 날카롭게 갈았고", 이제 "실제로 베어보는" 단계.**

## 1. compatibility.json 보강 방법론 (심도깊게)

### 1-1. 현실 진단 — 무엇이 어렵나
- 손으로 채운 57개는 **내가 직접 겪은 모델**뿐. 확장이 수동·느림.
- Manager 538개를 붙여도 최신·특수 모델(LTX2.3/Ideogram4/개인 파인튜닝)은 여전히 구멍.
- 즉 "정적 DB 키우기"는 **밑 빠진 독**. 모델은 매주 새로 나온다.

### 1-2. 핵심 통찰 — DB를 "채우는" 게 아니라 "자라게" 한다
정적 DB 완성은 불가능. 대신 **쓰면서 자동으로 쌓이는 구조**가 현실적이다.
이미 4층 구조(curated→Manager내장→Manager실시간→web_search)가 그 토대.
여기에 **"검색 결과를 큐레이션 DB로 승격"** 루프를 더하면 자생한다:

```
사용자가 "이 모델 검색"(4층 web_search) 누름
  → 실제 HF URL 찾음
  → [신규] "이거 맞으면 저장" 버튼
  → compatibility.json 후보로 적립 (양자화·VRAM·폴더까지)
  → 다음 사용자는 1층(큐레이션)에서 바로 뜸
```
→ 한 번 검색한 건 두 번 검색 안 함. **사용할수록 DB가 두꺼워진다.** 이게 "BUILT, NOT JUST GENERATED"의 진짜 구현.

### 1-3. 보강 경로 4가지 (현실성 순)
1. **자동 적립 (위 루프)** — 가장 현실적. 검색 1회 = DB 1개 성장. 수동 입력 0.
   - 단 "저장"은 사용자 확인 후(할루시네이션 방지). 로컬에서만(키 있을 때).
   - 적립 형식: 후보 JSON으로 쌓고, 주기적으로 사람이 검수해 본 DB 병합.
2. **양자화 룰 확장 (데이터 아닌 룰)** — 모델이 아니라 "형식 규칙"을 늘림.
   - 지금: fp8/fp4/mxfp8/gguf/bf16. 추가 가능: int4/int8, nf4, awq, gptq, Q4_K_M 세분, fp8_e5m2 등.
   - 룰은 1개 추가 = 무한 모델 커버. **데이터보다 ROI 압도적.** ★우선순위 높음.
   - GPU 세대도: Ada FP8 네이티브, Blackwell FP4 네이티브, Hopper 등 더 정밀히.
3. **워크플로별 "알려진 지뢰" 축적** — 모델 URL이 아니라 "이 파이프라인 = 이 함정" 지식.
   - 예: "LTX2.3 = gemma fp4를 Ampere에서 못 씀", "Trellis2 = spconv 컴파일", "Wan = umt5 fp8".
   - 이건 네가 직접 겪은 tribal knowledge라 공개 DB에 없는 차별점. niche 깊이.
4. **커뮤니티 PR (먼 미래)** — 오픈소스화하면 남이 채워줌. 단 사용자 수 적어 현실성 낮음. 보류.

### 1-4. 결론 — 보강 우선순위
- **단기**: 양자화 룰 확장(2) — 데이터 0, 커버리지 ↑↑. 지금 바로 가능.
- **중기**: 자동 적립 루프(1) — 검색→저장 버튼. 쓸수록 자람.
- **장기**: 지뢰 지식 축적(3) — 실증하며 자연히 쌓임(아래 §2와 맞물림).

## 2. 실증 계획에 대한 전문가 의견 (사용자의 PC 검증)

### 2-1. 이 계획이 옳은 이유 (강하게 동의)
> "배포 후 내 PC에서 실제 빨간줄 오류를 티어다운으로 풀어본다. LLM 바로 안 가고,
>  티어다운 정보로 파일도 받아보고 실제 해결되는지 본다."

이건 **제품 검증의 정석**이다. 이유:
- 도구의 진짜 가치는 "내가 만들었으니 좋다"가 아니라 "실제 막힘을 풀었나"로만 증명됨.
- 합성 테스트(우리가 한 것)는 한계가 명확 — 실데이터가 진실을 말함.
- "LLM 바로 안 간다"가 핵심. 티어다운이 LLM보다 **먼저·빠르게** 가치를 주는지가 시험대.
- 풀다가 막히는 지점 = 도구의 진짜 약점 = 다음 개선 1순위. **실패가 가장 값진 데이터.**

### 2-2. 검증을 제대로 하는 법 (전문가 제안)
그냥 "써본다"보다, **구조화된 기록**을 남기면 검증이 포트폴리오 자산이 된다:

| 측정 항목 | 왜 |
|---|---|
| 워크플로별: 티어다운이 **몇 %를 풀었나** | 도구 커버리지 정량화 |
| 티어다운이 준 직링크로 **실제 파일 받아 작동했나** | 직링크 정확도(B효용 검증) |
| 양자화 경고가 **실제 막힘과 일치했나** | 핵심 기능 정확도 |
| 티어다운으로 **안 풀려 LLM 간 것** | 도구 한계 = 개선 1순위 |
| 티어다운 → LLM 브리핑이 **시간 단축했나** | 컨텍스트 조립 가치 입증 |

→ 이 표를 케이스마다 채우면, "내 도구로 N개 워크플로 중 X% 자력 해결" 같은 **숫자**가 나온다.
   포트폴리오에서 "만들었다"가 아니라 "검증했다"로 격상. 면접 무기.

### 2-3. 검증 중 나올 법한 약점 (미리 대비)
- 직링크가 틀릴 수 있음 → "확인 필요" 정직 표기가 이때 빛남(거짓말 안 함).
- 폴더 경로가 모델마다 다름 → save_path 정확도 점검 필요.
- 양자화 경고가 과탐/미탐 → 룰 미세조정 데이터 확보.
→ 이 약점들이 바로 다음 작업 목록이 됨. **검증이 곧 로드맵 생성기.**

## 3. 이후 해볼 수 있는 것들 (가능성 확장)

### 3-A. 도구 자체 강화 (실증에서 약점 나오면)
- **양자화 룰 대폭 확장** (§1-3의 2) — int4/nf4/awq/gptq 등. 데이터 없이 커버 ↑.
- **검색→DB 자동 적립** (§1-2 루프) — 쓸수록 자라는 구조.
- **폴더 배치 정확도** — 같은 모델도 워크플로마다 다른 폴더. 노드 타입(UNETLoader→diffusion_models)으로 추론 정밀화.
- **C효용: 설치 스크립트** (이번에 설계) — .sh/.bat 한 방 실행.

### 3-B. 검증 결과 활용
- **실증 케이스 = 포트폴리오 서사** — "이 워크플로가 안 돌았다 → 티어다운이 짚었다 → 풀었다" 3막.
- **실패 케이스 = 정직한 한계 인정** — "이건 티어다운도 못 풀어 LLM 갔다"가 오히려 신뢰.

### 3-C. 더 큰 그림 (장기, 선택)
- **회사 내부 도구화** — BLOT C&M ComfyUI 파이프라인 표준 진단기로. (사용자 업무와 직결)
- **ComfyUI 확장(custom node)으로 이식** — 웹 도구 대신 ComfyUI 안에서 직접. 단 큰 작업.
- **양자화 호환성만 떼어 특화** — "내 GPU에서 이 모델 돌아가?" 단일 기능 미니 도구. 명확한 니즈.

### 3-D. 우선순위 제안
1. **(지금) C효용 설치 스크립트** — PRD 마지막 퍼즐.
2. **(배포 후) 실제 PC 실증** — 사용자 계획. 도구 진가 측정.
3. **(실증 후) 약점 기반 개선** — 양자화 룰 확장 / 폴더 정확도 / 자동 적립.
4. **(여유되면) 포트폴리오 서사 정리** — 실증 케이스로.

---

## 4. 핵심 메시지
- comp.json은 "완성"하는 게 아니라 "자라게" 한다 (자동 적립 + 룰 확장).
- 데이터보다 룰이 ROI 높다 (1룰 = 무한모델).
- 실증이 다음 로드맵을 만든다 (실패가 가장 값진 데이터).
- 가시적 결과는 "내 PC에서 실제로 풀었다"에서 나온다.
