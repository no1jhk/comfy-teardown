# Teardown v2 방향 확정 — "처음 실행 지옥" 해결기

> 2026-06-22~26 결정. 다음 세션 시작 시 이것부터 읽기. 방향이 또 흔들리지 않게 못박는 용도.

---

## 0. 목적 (한 줄, 못박음)

**"누구나 받은 새 JSON을 처음 실행할 때, 모델·노드 다 받았는데도 꼭 안 되는 게 생긴다. 그 '처음 실행 지옥'을 끝내주는 도구."**

- 내 문제(3060↔3090 재현)가 아니라 **모든 ComfyUI 입문자의 공통 문제**.
- 범용 지향. 단 "마찰 있는 워크플로"(LTX2.3~Trellis급)가 실제 대상. 진짜 가벼운 건(Ideogram4) 효용 낮음(어차피 안 막힘).

---

## 1. A vs B — 무엇을 푸는가

| | 내용 | 판정 |
|---|---|---|
| A | "이 모델 어디서 받아?" → 정확한 직링크 | 곁다리. 외부 권위 데이터로만 (손으로 DB 안 만듦) |
| B | "다 받았는데 왜 안 돼?" → 환경/설정 진단 | ★ 궁극 목적. 여기 집중 |

---

## 2. 3층 구조 (확정)

```
[A층: 받기] ← 외부 권위 데이터 (손으로 안 만듦)
  1순위. 워크플로 JSON 자체 메모/메타 (직링크 박힌 경우)
  2순위. ComfyUI Manager 맵 (model-list.json, extension-node-map.json) — 방대·권위·무료
  3순위. web_search 폴백 (위에 없을 때만)

[B층: 돌아갈까=호환성] ← 손으로 큐레이션 = moat
  torch/cuda/flash_attn/VRAM/fp8-Ampere 호환성
  소스: 내 삽질 경험 + 개발자 compatibility.json + README

[폴백: 새 문제] ← LLM 추론
  룰이 못 잡는 새 에러 → 완성 브리핑을 LLM에 (비용 0, 붙여넣기)
  → 풀면 룰로 박제 → 다음부턴 무료
```

---

## 3. 도구 본질 (흔들림 방지)

- ❌ "모든 새 문제를 푸는 해결사" (불가능 — LLM도 새 건 모름)
- ✅ **"누군가 한 번 겪은 문제는 두 번 다시 안 겪게 하는, 경험이 쌓이는 도구"**
- 룰 = 흔한 것 즉시·무료 / LLM = 새 것 1차 시도 / 내 삽질 = 원천 데이터

---

## 4. 안 할 일 (방어선)

- ❌ 직링크 정적 DB를 손으로 구축 (Manager가 이미 함 → A층은 가져다 씀)
- ❌ "노드 찾기·설치"로 Manager와 경쟁
- ❌ "단발 에러 빠른 진단"으로 LLM과 경쟁
- ❌ 자동 다운로드 (웹기반이라 보류)
- → 이 셋으로 새면 "설 자리 없음" 반복. B(환경 진단)에 집중.

---

## 5. 핵심 피칭

> "디버깅의 병목은 AI의 답이 아니라, AI에게 줄 컨텍스트의 조립이었다. 그래서 그 조립을 자동화했다."

→ "LLM 래퍼 아니냐" 방어선. Bernini(Wan2.2) 테스트로 범용성 실증됨.

---

## 6. CWAP (회사 개발자 도구)와의 관계

- CWAP = ComfyUI Workflow Auto-Provisioner. .exe + compatibility.json. JSON→환경감지→설치/대체/우회까지.
- 같은 문제를 더 완성도 높게(설치형) 품. compatibility.json = 노드별 torch/cuda/build_deps/fallback + 모델 57개 직링크+VRAM+폴더.
- **관계: 경쟁 아니라 보완.** CWAP=설치형(그 PC에서 실행), Teardown=웹(무설치 사전점검+디자인+공유).
- compatibility.json을 Teardown의 B층 데이터로 활용 (개발자가 나 도우려 만든 것).
- 교훈: "DB 불가"는 틀린 판단이었음 — 범용 가정이라 그랬고, 헤비 범위면 10노드 규모로 충분. "직접 해본 사람이 추론만 하는 AI를 이긴다 = BUILT NOT GENERATED".
