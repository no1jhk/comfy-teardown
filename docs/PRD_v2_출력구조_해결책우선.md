# Teardown — 출력 구조 재설계: "해결책 먼저, 원인은 뒤에" (가독성 우선)

> VS Code Claude Code용. 대상: src/Teardown.jsx
> 사용자 핵심 지적: "사용자는 왜 안 되는지보다 뭘 하면 되는지를 먼저 원한다.
>  해결책을 표로 순번대로. 그 후 원인·환경설정을 한눈에."
> → 두세 번 점검한 구조.

---

## 0. 문제 진단 (두세 번 점검)

### 현재 무엇이 거꾸로인가
1. **buildBriefing이 LLM에 출력 형식을 안 정해줌** → "단계별로 알려주세요"만 함
   → LLM이 자기 맘대로 원인분석부터 장황하게 씀. ★이게 사용자가 겪은 "너무 장황" 의 직접 원인.
2. **Solution 카드도 step 순서가 "원인 위주"** → 양자화 경고(왜) 다음에 설치(무엇)가 옴.
   → 사용자는 "무엇을 하면 되나"(설치·교체·다운로드)를 맨 위에서 보고 싶음.

### 핵심 원칙 (재설계 기준)
- **TL;DR 먼저**: "딱 이것만 하면 됨" 3~5줄을 맨 위에.
- **따라하기 표**: 순번 + [무엇을] + [어디에/어떻게] 표. 복붙·체크 가능하게.
- **원인은 접어서 뒤에**: "왜 이게 문제였나"는 펼쳐보기(아코디언)로 아래.
- **환경 한 줄 요약**: "이 PC에선 이 설정" 한눈에.

---

## 1. buildBriefing 개선 (★ 우선순위 1 — LLM 출력 형식 강제)

LLM이 장황한 진짜 원인 = 출력 형식 미지정. **브리핑 맨 앞에 "이 형식으로 답하라"를 명시.**

현재 (함수 앞부분):
```js
L.push(`아래는 ComfyUI 워크플로의 구조 분석 + 환경 + (있으면)에러 로그입니다.`);
L.push(`이 정보를 종합해 이 워크플로의 구체적 노드·모델·환경에 맞춰 진단하고, 해결을 단계별로 알려주세요.`);
L.push(`확신 없는 부분은 솔직히 밝히고, 없는 URL·파일을 지어내지 마세요.`);
```
변경 (출력 형식 지정 추가):
```js
L.push(`아래는 ComfyUI 워크플로의 구조 분석 + 환경 + (있으면)에러 로그입니다.`);
L.push(`이 정보로 진단하되, 반드시 아래 "출력 형식"을 지켜 답하세요.`);
L.push(``);
L.push(`### 출력 형식 (반드시 이 순서)`);
L.push(`1. **해결 요약** (3~5줄): 지금 당장 뭘 하면 되는지만. 원인 설명 없이 행동만.`);
L.push(`2. **해결 단계 표**: | 순번 | 할 일 | 대상(파일/노드) | 어디에/어떻게 | 형식의 표. 따라하기 쉽게.`);
L.push(`3. **이 PC 환경 설정** (있으면): 이 GPU/torch/CUDA에서 바꿔야 할 설정 한눈에.`);
L.push(`4. **원인 설명** (맨 뒤, 짧게): 왜 이 문제가 생겼는지. 1문단 이내. 장황 금지.`);
L.push(``);
L.push(`규칙: 해결을 먼저, 원인은 맨 뒤 짧게. 확신 없으면 솔직히, 없는 URL·파일은 지어내지 마세요.`);
```
- → 이러면 LLM이 표·순번·해결우선으로 답함. **장황함 해결의 핵심.**

## 2. Solution 화면도 같은 구조로 (도구 자체 출력)

### 2-A. TL;DR 블록 추가 (Solution 맨 위)
처방(rx) 단계들 위에, **"딱 이것만" 요약 박스**:
```
[요약] 이 워크플로를 돌리려면:
· 양자화 2개 교체 (fp8→GGUF)
· 노드 4개 설치
· 모델 8개 배치 (1개 자동, 7개 수동)
```
- rx 단계들의 title을 1줄씩 압축. C.point 테두리 박스, 굵게.
- 사용자가 스크롤 안 해도 "뭘 해야 하는지" 한눈에.

### 2-B. 해결 단계를 "표"로 (순번 + 할일 + 대상 + 방법)
현재 step 카드(접이식)는 유지하되, **모델 다운로드 step은 표 형태**로:
| # | 모델 | 폴더 | 받기 |
|---|---|---|---|
| 1 | ltx-2.3-22b-dev.safetensors | models/checkpoints | [다운로드] (확정) |
| 2 | gemma_3_12B_fp4.safetensors | models/checkpoints | [이 모델 검색] (미확인) |
- 표로 하면 "뭘 어디에"가 한눈에. 복붙·체크 가능.
- 확정=다운로드 버튼/링크, 미확인=검색 버튼. 색으로 구분.

### 2-C. 원인은 접어서 뒤로
양자화 경고 카드(severity high)는 유지하되, **상세 원인 설명은 기본 접힘**.
- 제목만 보임: "양자화 비호환 2건 — GGUF로 교체" (행동)
- 펼치면: "왜 fp8이 Ampere에서 안 되나" 설명 (원인)
→ 행동이 먼저, 원인은 펼쳐야 보임.

## 3. .md 저장(buildMarkdown)도 같은 순서
buildMarkdown도 해결요약 → 단계표 → 환경 → 원인 순으로 재배치. (브리핑과 동일 구조)

---

## 우선순위 (이번에 다 안 해도 됨)
1. **buildBriefing 출력 형식 지정** (★ 제일 효과 큼. LLM 장황함 직접 해결). 작음.
2. **Solution TL;DR 블록** (2-A). 중간.
3. **모델 다운로드 표** (2-B). 중간.
4. 원인 접기(2-C), .md 재배치(3). 작음~중간.

→ 1번만 해도 "LLM 답 장황함"은 크게 개선. 2~4는 도구 자체 화면 개선.

## 완료 기준
1. LLM 분석 프롬프트에 "출력 형식(해결요약→표→환경→원인)" 지시가 포함됨.
2. (선택) Solution 맨 위 TL;DR 요약 박스.
3. (선택) 모델 다운로드가 표 형태.
4. npm run build 에러 없음. 회귀 없음.
