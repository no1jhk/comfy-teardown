# PRD v1.1: Teardown — AI 진단 레이어

> v1.0(룰 기반 진단)은 완성·배포됨. v1.1은 그 위에 **LLM 레이어**를 얹어
> "진단까지"에서 "실행 가능한 처방까지"로 효용을 끌어올린다.
> v1.0의 report 객체(구조 분석 결과)를 입력으로 받아, LLM이 그걸 맥락으로 해석한다.

---

## 0. 왜 v1.1인가 (해결할 격차)

v1.0의 정직한 한계 — 사용자가 직접 지적한 지점:
- 모델 HuggingFace 연결이 대부분 **검색 링크만** (정확한 다운로드 주소 아님)
- 출처가 이름 패턴 기반 **"추정"** (확정 아님)
- 에러 진단이 **룰 11패턴 한정** (그 밖은 "패턴 못 찾음")

→ 결과적으로 "진단은 하지만 해결은 사용자에게 떠넘긴다."

**v1.1 목표 (효용 기준선):**
> "개발자가 의도한 것처럼, 알아서 설치까지는 아니어도 **그 바로 밑까지**는 해준다."
> 즉 *이 링크 클릭 → 이 파일 여기 넣어 → 이 명령 실행* 까지 손에 쥐여주는 수준.
> 사람이 한 줄씩 손대지 않아도 되는 처방.

이 효용이 입증돼야 v1.1이 성공이다. (PRD §7 v4 "맞춤 설치 스크립트"가 원래 비전 — v1.1로 당김)

## 1. 세 가지 효용 (모두 LLM 레이어)

### 효용 A — 에러 로그 → LLM 맥락 진단 (임의 에러 해석)
- 현재: 룰 11패턴. 패턴 밖은 "못 찾음".
- v1.1: 룰이 1차로 빠르게 돌고, **못 잡거나 사용자가 "AI 정밀 진단" 누르면 LLM 2차.**
- LLM 입력: 에러 로그 전문 + v1.0 report(노드 구조·pack·이식위험) + (선택) 캡처 이미지.
- LLM 출력: 이 워크플로의 **구체 노드와 결합한** 원인·처방.
  - 예: "당신의 HYMotionDiTLoader 노드 + 이 size mismatch = hyMotionLite.ckpt가 Lite가 아닌 Full 버전일 가능성. Inventory의 그 파일을 Lite로 교체."
- **하이브리드 원칙**: 룰=빠르고 공짜(1차), LLM=느리지만 유연(2차 fallback/심화).

### 효용 B — "추정" → "확정" (모델·repo 실제 검색)
- 현재: `guessFolder()`로 "추정", HF는 검색 링크만.
- v1.1: LLM이 **웹 검색**으로 실제 repo·다운로드 URL·파일 크기·정확한 배치 폴더를 찾고 검증.
- 출력: "추정" 딱지 → "확인됨(출처: …)"으로 승격. 못 찾으면 정직하게 "확인 필요" 유지.
- **검증 레이어 필수**: LLM이 URL을 지어내지 않도록, 검색 결과에 실재하는 것만. 출처 표기.

### 효용 C — 실행 가능한 처방 (설치 스크립트 생성) ★ 핵심 효용
- 현재: "git clone 하세요" (어디서·어떻게는 사용자 몫).
- v1.1: 이 워크플로 **전용 설치 스크립트 한 방** 생성.
  - 노드 N개 clone + 모델 M개를 **확정된** 위치에 다운로드 + 환경 우회(flash_attn→sdpa 안내).
  - OS별 `.sh`(Mac/Linux) / `.bat`(Windows) 두 버전.
  - 복사 → 한 번 실행 → 끝. 사람이 한 줄씩 손댈 필요 없는 수준.
- 이게 "바로 밑까지"의 실체. A·B의 결과(확정된 repo·URL)가 있어야 C가 신뢰성 있음.

## 2. 신뢰 설계 (할루시네이션 방지) — "BUILT, NOT JUST GENERATED"

LLM 붙이면 최대 리스크 = 그럴듯한 거짓말. 막는 장치:
- **검색 출처 동반 표기.** URL·repo는 검색 결과에 실재하는 것만. 출처 링크 같이 노출.
- **확신 없으면 "확인 필요"로 정직하게 남김.** 억지로 확정하지 않음.
- **룰 우선, LLM은 보강.** 룰이 답하는 건 LLM 안 부름(비용·속도·신뢰).
- **LLM 답을 그대로 뱉지 않음.** 구조화(JSON) → 검증 → 렌더. raw 출력 직접 노출 금지.
- 사용자가 진단 근거를 추적 가능해야 함(왜 이렇게 진단했는지).

## 3. 기술 구조 (어떻게 붙이나)

### 3-A. API 호출 방식 — 청중 3단계에 따라 다름 (★ 비용·키 핵심)

결정적 사실: claude.ai 아티팩트 환경은 `fetch("https://api.anthropic.com/v1/messages")`를
키 없이 호출 가능. **하지만 이건 claude.ai 안에서만.** Vercel에 배포하면 이 특수 환경이
없어 호출이 막힌다. → 배포본에서 실제 LLM을 쓰려면 키·비용 문제가 반드시 생긴다.

청중이 셋(나 → 회사 → 포트폴리오)이고, **각 단계의 정답이 다르다:**

| 단계 | 청중 | 방식 | 키 | 비용 |
| --- | --- | --- | --- | --- |
| 1 | 나 (효용 검증) | **아티팩트 프로토타입** (claude.ai 내) | 불필요 | 0 |
| 2 | 회사 (능력 시연) | **로컬 `npm run dev` + `.env.local`에 본인 키** | 본인 키, 로컬만 | 푼돈(통제) |
| 3 | 포트폴리오 (누구나) | **배포본: 룰=라이브 / AI=데모 모드** | 불필요 | 0 |

**1단계 (나):** 아티팩트가 그 용도. claude.ai 안에서 키 없이 즉시 작동. 추가 작업 없음.

**2단계 (회사 시연):** 로컬 실행이 정답. `.env.local`에 본인 Anthropic 키 넣고
`npm run dev`로 맥북에서 띄워 직접 시연/화면공유. 키 외부 노출 0, 비용 통제(시연 몇 번).
→ "BUILT, NOT JUST GENERATED"를 옆에서 직접 돌려 보여주는 게 임팩트도 큼.
(LottoMetis·Quant Metis에서 쓰던 `.env.local` 방식 그대로. `VITE_ANTHROPIC_API_KEY` 등.)

**3단계 (포트폴리오, 누구나 접근):** 실제 LLM 호출을 **하지 않는다.**
- 룰 기반 v1.0 = 풀 라이브(지금처럼 완전 작동).
- AI 부분 = **"데모 모드"**: 미리 준비한 진단 결과 1~2개를 보여주는 버튼. 실제 호출 X.
- 작게 표기: "실시간 AI 진단은 로컬/시연 버전. 코드는 GitHub 공개."
- 이유: 모르는 사람이 무한정 호출 → 본인 청구서 폭탄. 채용 담당은 "작동+설계"만
  확인하면 되지 자기 돈으로 LLM 돌릴 필요 없음. 데모 모드면 충분하고 가장 안전.

**핵심 원칙: 배포본(누구나)에서는 실제 LLM 호출을 안 하는 게 정답.**
진짜 호출은 1단계(아티팩트)·2단계(로컬)에서만. → 키 노출·비용 폭탄 원천 차단.

참고로 "배포본 + 본인키 Serverless 프록시 + 사용량 제한"도 가능은 하나(아래),
포트폴리오 단계에선 데모 모드가 더 안전·단순해 우선하지 않는다.
- (대안, 미채택) Vercel Serverless Function에 본인 키를 env로 숨기고 일 호출 N회 제한.
  링크만 보내면 되지만 설정 부담·비용 노출 리스크가 남음.

### 3-B. 모델 선택
- `claude-sonnet-4-6` (균형). 정밀 진단은 필요시 상위 모델 검토.
- 웹 검색 도구(web_search) 결합 → 효용 B(실제 repo·URL 검색)에 필수.

### 3-C. 데이터 흐름
```
[v1.0 report 객체]  ← 이미 있음 (구조 분석 결과)
        +
[에러 로그 / 캡처]  ← 사용자 입력
        ↓
[프롬프트 조립]  ← report를 컨텍스트로, "이 워크플로 한정" 진단 지시
        ↓
[Claude API + web_search]
        ↓
[JSON 구조화 출력]  ← {진단[], 확정모델[], 스크립트{sh,bat}}
        ↓
[검증 레이어]  ← URL 실재 확인, 확신도 체크
        ↓
[렌더]  ← 기존 UI 톤 유지. "AI 진단" 섹션 / "확인됨" 승격 / "스크립트 복사" 버튼
```

### 3-D. report 객체 재사용
- v1.0이 만든 `report`(packs/unmapped/models/portability/muted…)를 그대로 LLM 컨텍스트로.
- 설계 원칙(PRD §9) "report 표준 객체 하나로 통합"이 여기서 보상받음 — 추가 파싱 불필요.

## 4. UI 통합 (기존 화면에 얹기)

- **효용 A**: 기존 "Error Diagnosis" 섹션에 "AI 정밀 진단" 버튼 추가.
  룰 진단 카드 아래, 못 잡았거나 더 깊이 원할 때 누름 → LLM 카드 추가.
- **효용 B**: Inventory 모델 카드의 "찾아보기(추정)"가 검증 후 "다운로드(확인됨)"로 승격.
  검증 중 로딩 상태 표시.
- **효용 C**: Solution 섹션 하단에 "설치 스크립트 생성" 버튼 → 생성된 .sh/.bat을 코드블록+복사+다운로드.
- 톤·색·컴포넌트는 v1.0 그대로(C 토큰). 새 섹션도 기존 박스·아코디언 패턴 재사용.
  → **이래서 컴포넌트 추출 리팩터링(AUTOPILOT §R)이 v1.1 전에 되면 좋다.**

## 5. 단계별 구현 순서 (검증 우선)

전체를 한 번에 X. 효용을 **가장 빠르게 입증되는 것부터.**

1. **[A 먼저] 에러 → LLM 맥락 진단** (아티팩트 버전)
   - 가장 데모 임팩트 큼. report + 에러 로그 → 구조화 진단 1개 띄우기.
   - 성공 기준: 룰이 못 잡는 실제 에러를 넣었을 때, 이 워크플로 노드와 결합한 처방이 나옴.
2. **[B] 모델·repo 검색·확정**
   - web_search 결합. 샘플 워크플로의 모델 5개 중 몇 개가 "확정"으로 승격되나.
   - 성공 기준: clip-vit-large-patch14 등이 실제 HF repo URL로 확정됨(출처 표기).
3. **[C] 설치 스크립트 생성**
   - A·B 결과(확정 repo·URL) 위에서 .sh/.bat 조립.
   - 성공 기준: 생성된 스크립트가 (검토상) 실제로 돌아갈 형태. 가짜 URL 없음.
4. **[배포] Vercel 프록시** — 효용 입증된 뒤에. 아티팩트→배포 전환.

## 7. 완료 기준 (이게 되면 v1.1 끝)

깨진 워크플로 JSON + 실제 빨간 에러 로그를 넣었을 때:
1. 룰이 못 잡는 에러도 **이 워크플로 노드와 결합한** 구체 처방이 나온다. (효용 A)
2. 모델 몇 개가 "추정"에서 **실제 URL 확정**으로 승격된다(출처 표기). (효용 B)
3. **복사해서 한 번 실행하면 되는 설치 스크립트**가 생성된다(가짜 URL 없음). (효용 C)
→ "이것만 보고 실제로 워크플로를 고칠 수 있다"는 효용이 입증된다.

## 8. 리스크 & 대응

| 리스크 | 대응 |
| --- | --- |
| LLM이 가짜 URL·repo 지어냄 | web_search 결과에 실재하는 것만, 출처 표기, 못 찾으면 "확인 필요" 유지 |
| 배포 환경에서 API 직접 호출 불가 | 1차는 아티팩트로 효용 입증 → 2차 Vercel 프록시 |
| API 비용 | 룰 우선(LLM은 fallback/심화만), 사용자 키 입력 옵션 검토 |
| 응답 느림(검색 포함) | 로딩 상태 명확히, 룰 결과 먼저 보여주고 LLM은 비동기 추가 |
| 정직성 훼손(억지 확정) | "확인 필요" 유지를 실패가 아닌 정상 출력으로 설계 |
