# PRD: Teardown — ComfyUI 파이프라인 진단 도구

## 1. 한 줄 정의
무거운 ComfyUI 워크플로(Trellis2·UniRig·HYMotion) JSON을 해체해, 필요한 노드·모델·패키지와
버전 충돌·이식 위험을 진단하는 웹 도구. (설치는 사람이, 진단·안내는 도구가.)

## 2. 문제 (왜 만드나)
- Trellis2+UniRig+HYMotion 류는 cuda/torch/python·flash_attn 충돌로 설치 지옥.
  본인이 며칠간 LLM 왕복하며 빨간 줄 캡쳐→질문→수정 반복으로 겨우 해결.
- 매번 손으로 한 일이 동일: pack 찾기 → requirements 모으기 → 에러와 묶어 질문.
  → 그 컨텍스트 수집과 진단 루프를 자동화한다.
- 일반 미씽노드는 ComfyUI-Manager가 이미 해결 → 이 도구의 대상이 아님.
  Manager가 못 푸는 "컴파일 지뢰 파이프라인"에 집중(포지셔닝).

## 3. 핵심 가치 (차별점)
- LLM 호출이 아니라 "컨텍스트 자동 조립 + 알려진 지뢰 사전 경고"가 가치. (ChatGPT 래퍼 아님)

## 4. 타겟 사용자
- 유튜버 워크플로 JSON 받아 무거운 파이프라인 돌리려다 막힌 사람 (본인 포함)

## 5. 이번 버전(v1.0 MVP) 범위 — 이것만 만든다
- [x] JSON 업로드 (드롭존 / 파일선택 / 샘플)
- [x] UI·API 포맷 정규화 + node class_type 추출
- [x] 내장/커스텀 분류 + cnr_id·prefix → 저장소 매핑(역인덱스 시드)
- [x] 프론트엔드 전용 노드 제외(오탐 방지)
- [x] 버전 충돌 흔적 검출 + 같은 저장소 복수 cnr_id 통합 안내
- [x] 이식 위험 값 lint (경로 구분자 \ / flash_attn / 과거 산출물)
- [x] 모델·자산 인벤토리 + 파일명 리네임 힌트
- [x] 비활성(muted/bypass) 노드 lint

## 6. 이번 버전에서 "안 하는 것" — 명시적 제외
- AI 진단(컨텍스트 자동 조립 → LLM) → v1.1 (API 키·CORS 프록시 필요)
- 자동 설치 실행 / 충돌 자동 해결 (사람이 복붙)
- 로컬 ComfyUI 연동
- 동작 환경 스냅샷 저장 (→ v2)

## 7. 다음 버전 로드맵 (말로만, 안 만듦)
- v1.1: AI 진단 — 분석 결과 + 에러 → LLM 구조화 진단 (백엔드 프록시)
- v2: 동작 환경 스냅샷 저장·불러오기(lockfile)
- v3: 설치 전 pack간 torch/python 충돌 사전 점검
- v4: 진단 → 맞춤 설치 스크립트 생성 (개발자 배치의 CUDA→torch 휠 선택 로직 차용)

## 8. 기술 스택
- React 18 + Vite 5, 브라우저 파싱(서버 불필요), Vercel 배포

## 9. 설계 원칙 (확장 대비)
- 파싱부(normalize) / 분석부(analyze) / 렌더부 분리
- 분석 결과는 표준 객체 하나(report)로 통합 → v2 스냅샷이 그대로 저장
- 매핑·룰은 상수 테이블(REPO_BY_CNR / REPO_BY_PREFIX / RENAME_HINT)로 분리 → 확장 쉬움

## 10. 완료 기준 (이게 되면 v1.0 끝)
- 깨진 워크플로 JSON 하나 올렸을 때 → pack·버전 / 출처 추정 / 이식 위험 /
  모델 목록 / 비활성 노드가 한 화면에 뜬다. ("샘플로 보기"로 즉시 검증 가능)

## 데모/포폴 서사
"ComfyUI 버전 호환성 지옥을 며칠간 수동으로 풀면서, 그 디버깅 루프가 자동화 가능함을 봤다.
개발자가 한방 설치 배치까지 만들어줬지만 flash_attn에서 막혔다 — 그 부류는 설치가 아니라
진단의 영역이라서. 그래서 컨텍스트 수집 + 사전 경고를 한 화면에 묶은 도구를 만들었다."
