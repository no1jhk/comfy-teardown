# ROADMAP.md — Teardown 엔지니어링 로드맵

> 외부 감사 2건에서 채택한 인프라·툴링·보안 개선 항목. 근거를 한 줄씩 병기한다.
> **제품 기능 로드맵(v1.1 AI 진단 ~ v4 설치 스크립트 생성)은 `docs/PRD.md §7` 참조** — 이 문서는 기능이 아니라 "어떻게 안전·견고하게 만드나"의 축이다.
> 우선순위: 단기(다음 사이클) · 중기 · 보류. 착수 시 이 문서 갱신 + HISTORY.md 기록.

## 단기

- **서버리스 프록시 (Vercel Functions · Anthropic 호출 서버 이전)**
  근거: 현재 AI 진단(v1.1)은 클라이언트에서 API 키를 못 쓴다(Vite `VITE_` 키는 빌드 번들에 평문 인라인 → 배포본 노출). 키·CORS를 서버 함수에서 처리해야 배포본에서 AI를 안전하게 켤 수 있다. (현행 `AI_KEY` 가드는 PROD 시 키를 비워 dev 전용 — 프록시가 이 제약의 정식 해소책.)

- **fixture ASCII 정상화**
  근거: `test/fixtures/`의 한글·유니코드 파일명이 CI 러너·타 플랫폼·셸에서 깨지거나 매칭 실패할 수 있다(현재 테스트는 `startsWith` 프리픽스 탐색으로 우회 중). ASCII 파일명으로 정규화하면 이식성·CI 안정성이 오른다.

- **Vite / esbuild 업그레이드**
  근거: esbuild가 JSX 구문 에러를 출력하고도 `exit 0`으로 넘겨 빌드 파손을 잠복시킨 사고 이력(HISTORY 5ec1a68). 최신 버전의 진단 정확도·보안 패치를 반영하고, 커밋 게이트(build 에러 부재 판정)의 신뢰도를 높인다.

- **GitHub Actions CI (build + smoke + e2e)**
  근거: 현재 `build`·`smoke`(3부)·`e2e`(15케이스)를 로컬 수동 실행에 의존한다. push마다 자동 게이트로 돌리면 회귀·TDZ·JSX 파손을 병합 전에 차단(런타임 크래시 사고 9fd2b07·356f5db 계열 예방).

## 중기

- **Teardown.jsx 분할 · Web Worker 검토**
  근거: 단일 컴포넌트가 2,900줄+로 커져 인라인 스타일 수술 시 사고 위험이 크다(간격·JSX 균형). 파싱(normalize)·분석(analyze)·렌더로 이미 논리 분리돼 있으니 파일 분할이 자연스럽고, 무거운 JSON 파싱은 Web Worker로 옮겨 대용량 입력 시 UI 블로킹을 없앨지 검토한다.

- **SECURITY.md**
  근거: 키 취급 원칙(가드·프록시), 배포 시 노출 위험, 취약점 제보 경로를 한 곳에 문서화(외부 감사 권고). 위 프록시 이전과 함께 정식화한다.

## 보류

- **조직형 확장 계열 (다중 사용자 · 팀 · 대시보드 등)**
  근거: 이 도구의 정체는 "빨간 워크플로우 JSON 한 번으로 환경 진단"인 단일·즉시 진단 도구(PRD §1·§4). 조직형 확장은 그 정체와 상이하며 스코프를 흐린다 → 채택 보류.
