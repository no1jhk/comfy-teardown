## 1. 점검 — PRD v1.1 세 효용 vs 실제 구현

PRD_v1.1에서 정의한 핵심 효용 3개 중 **2개가 누락**됨:

| 효용 | 기획 의도 | 현재 | 판정 |
|---|---|---|---|
| A. 에러→LLM 맥락 진단 | 룰 미해결 에러를 LLM이 워크플로 노드와 결합 진단 | 브리핑(4단계)+키있을때 AI진단 | △ 부분 |
| **B. "추정"→"확정"** | **web_search로 실제 HF URL 확정**, 못 찾으면 "확인 필요" | 정적 57개만. 모델 web_search 폴백 없음 | ❌ **누락** |
| **C. 설치 스크립트** | 복사→1회 실행 .sh/.bat 생성 | 미구현 | ❌ **누락** |

v2에서 양자화 진단(PRD에 없던 신규 가치)에 집중하느라 원래 기획 B·C를 건너뜀.

## 2. 검증된 사실 — 정적 DB로는 부족

실측(2026-06-27): 테스트 JSON 모델 매칭률
- 내 compat(57개): LTX 0/7, Bernini 0/5, Ideogram 0/5
- + Manager model-list.json(538개): LTX 1/7, Bernini 2/5, Ideogram 0/5

→ **정적 DB를 아무리 키워도 최신·특수 모델(LTX2.3/Ideogram4/개인 파인튜닝)은 못 따라잡음.**
→ PRD_v1.1의 B효용("web_search로 실제 검색해 확정")이 원래 정답이었음. 기획은 옳았고 구현만 누락.

---

## 3. 데이터 소스 3층 재설계 (확정)

모델 직링크 조회 순서 (교차검증):

```
1. 내 compatibility.json (57개)   ← 정밀 큐레이션 (양자화·VRAM·대안). 최우선
2. Manager model-list.json (538개) ← 광범위 직링크. 내장(A안). 2순위
3. web_search 폴백                 ← 위 둘에 없으면 실시간 검색(B효용). 최신 커버
```

- 1·2는 **교차검증**: 둘 다 있으면 일치 확인, 다르면 1 우선(큐레이션 신뢰).
- 3은 사용자가 "이 모델 검색" 누를 때(노드의 researchNode 패턴 모델 확장).
- web_search 결과는 출처 표기 + "확인 필요" 정직성 유지(PRD_v1.1 §2 할루시네이션 방지).

## 4. Manager model-list.json 갱신 대비책 (A안 내장의 숙제)

내장 방식이라 자동 최신화 안 됨 → 갱신 루틴 필요:
- **월 1회 점검** (매월 1일 또는 기억나는 날): scripts/update-model-list.mjs 실행
- raw URL: https://raw.githubusercontent.com/Comfy-Org/ComfyUI-Manager/main/model-list.json
- 스크립트가 받아서 src/data/manager-model-list.json 갱신 + 개수 비교 리포트
- 큰 변화 있으면 커밋. web_search 폴백이 있어 갱신 늦어도 치명적이지 않음(3층이 메움).
- HISTORY.md에 갱신일 기록.
